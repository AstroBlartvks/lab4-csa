## Принципы и общие решения

Следуем стилю референса:

- **Простота > элегантность.** Если можно без классов и без синтаксического сахара —
  пишем без. Никаких заводов по производству фабрик.
- **Минимум модулей, максимум плотности.** В референсе: `isa.py`, `translator.py`,
  `machine.py`, `golden_*_test.py` — **четыре** файла. Мы добавим только то, что
  требует вариант: `cache.py` (отдельно, чтобы тестировать), `microcode.py`
  (микропрограмма как данные).
- **DataPath / ControlUnit / Cache — пассивные классы**, методы которых соответствуют
  сигналам. Сигнал — атомарная операция, исполняется за один такт. Корректность
  использования сигналов — ответственность вышестоящего модуля.
- **Микрокод — данные, а не код.** Микропрограмма — это массив списков сигналов
  (как в OCaml-варианте референса). Каждый элемент — это микроинструкция, то есть
  список сигналов, которые надо одновременно подать на DataPath в этом такте.
- **Тестирование — golden tests** на YAML-конфигах + (опционально) doctest'ы
  внутри модулей для самых тонких мест (см. `signal_wr` в референсе).

### Ключевые архитектурные решения этого варианта

| Решение | Значение | Краткое обоснование |
|---|---|---|
| Машинное слово | **32 бита**, знаковое | хватает для адресов, значений, демонстрации 64-бит арифметики через пары |
| Стеки данных и возвратов | **на чипе**, отдельные SRAM по 256 слов | каждая инструкция использует стек; класть стек в основную (кешируемую) память — это сделать каждый `+` ценой 10+ тактов и убить демо кеша |
| TOS / NOS | отдельные регистры | бинарные ALU-операции работают за такт без обращения к SRAM стека; идиоматично для stack-машин (J1, MISC) |
| Кеш | direct-mapped, **4 строки × 4 слова**, write-back, write-allocate | минимальная схема, на которой видны hit/miss/eviction; write-back даёт интересную демонстрацию dirty-eviction |
| MMIO | **uncached**, регион `0xFFFFFFF0–0xFFFFFFFF` | избавляет от вопросов когерентности; ввод нельзя «закешировать» |
| Микрокод | **горизонтальный, как список сигналов**, в `signal list array` | стиль референса; каждый сигнал = метод DataPath/CU, исполняется за такт |
| ControlUnit | **mPC + декодер opcode → mPC**, без флагов «END_OF_INSTR» | по варианту: окончание инструкции = `LatchMpc Zero` (возврат к FETCH) |
| Условные переходы ISA | через сигнал `LatchPc Jz` | флаг zero вычисляется DataPath'ом, mux в CU выбирает Jmp/Next по нему |

## Язык программирования (Forth, подмножество)

### Синтаксис (EBNF)

```ebnf
program     ::= { token } ;
token       ::= number | string_lit | dot_string
              | colon_def | word_use | comment
              | char_lit | tick ;
number      ::= [ "-" ] digit { digit }
              | "0x" hexdigit { hexdigit } ;
string_lit  ::= 'S" '   { char_except_quote } '"' ;
dot_string  ::= '." '   { char_except_quote } '"' ;
colon_def   ::= ":" name { token } ";" ;
word_use    ::= name ;        (* любое слово из словаря *)
comment     ::= "(" { char_except_rparen } ")"
              | "\" { char } end_of_line ;
char_lit    ::= "[char]" name ;
tick        ::= "'" name ;    (* push execution token *)
```

Whitespace отделяет токены; иной структуры нет. **Это и есть Forth — слова
вызываются в порядке записи, всё через стек.**

### Семантика (кратко)

- **Стратегия вычислений** — строгая, слева направо.
- **Область видимости** — глобальная. Все определения видны после момента объявления.
- **Типизация** — динамическая, фактически бестиповая: всё на стеке — 32-битное слово.
- **Литералы:**
    - числа компилируются в `LIT <value>`, значение хранится в **следующем слове** памяти инструкций (так делается, потому что 32 бита значения не помещаются в operand-поле инструкции);
    - строки `." ..."` и `S" ..."` размещаются в data-секции как `cstr` (null-terminated, **1 символ = 1 машинное слово** — требование варианта); `." ..."` компилируется в push-адреса + `CALL` рантайм-функции `print-cstr`.
- **Процедуры:** `: name ... ;` компилируется в кусок кода, заканчивающийся `RET`. Имя слова в словаре связано с адресом первой инструкции — это и есть **execution token (xt)**. `' name` пушит xt на стек, `execute` снимает xt и делает CALL по нему.

### Минимальное подмножество слов (built-ins)

| Категория | Слова |
|---|---|
| Стек данных | `dup drop swap over` |
| Стек возвратов | `>r r> r@` |
| Арифметика | `+ - * / mod` |
| Сравнения | `= < > <> 0=` |
| Логика | `and or invert` |
| Память | `@ !` |
| Управление | `if ... else ... then`, `begin ... until` |
| I/O | `key emit cr` |
| Определения | `: ... ;`, `variable`, `constant` |
| Строки | `." text"`, `S" text"`, `[char] X` |
| Execution token | `' word`, `execute` |

Этого достаточно для prob1 (Euler #4: палиндром-произведение), для всех обязательных
тестов и для демонстрации специфики Forth (xt + execute).

## Организация памяти

Фон Нейман: единое 32-битное адресное пространство, **пословная адресация** (адрес N
указывает на слово N).

```text
       Адресное пространство (Neumann)
+------------------------------+ 0x00000000
| 0: JMP main                  |  ← стартовый прыжок
| 1: <тело слова foo>          |
| ...                          |
| n: RET                       |
|                              |
| <тело слова bar>             |
| ...                          |
+------------------------------+
| main:  ...                   |  ← точка входа
|        HALT                  |
+------------------------------+
| <data section>               |
| строка "Hello\0" (6 слов)    |  ← cstr
| variable x  (1 слово)        |
| ...                          |
+------------------------------+
| heap (свободно)              |
| ...                          |
+------------------------------+ 0xFFFFFFF0
| MMIO (uncached)              |
|  0xFFFFFFF0  IN_PORT (R)     |  ← read = pop из input буфера
|  0xFFFFFFF1  IN_STATUS (R)   |  ← 1 = есть данные, 0 = EOF
|  0xFFFFFFF2  OUT_PORT (W)    |  ← write = push в output буфер
+------------------------------+ 0xFFFFFFFF
```

### Стеки (на чипе, не в основной памяти)

```text
   Data Stack                Return Stack
+--------------+           +--------------+
| TOS register |           | TORS register|
+--------------+           +--------------+
| NOS register |
+--------------+
| DS SRAM:     |           | RS SRAM:     |
|   256 слов   |           |   256 слов   |
+--------------+           +--------------+
| DSP (8 бит)  |           | RSP (8 бит)  |
+--------------+           +--------------+
```

- `TOS`, `NOS` (data stack) и `TORS` (return stack) — отдельные регистры, чтобы
  бинарные операции брали два операнда за такт без чтения SRAM стека.
- При `dpush`: `SRAM[DSP] ← NOS`, `DSP++`, `NOS ← TOS`. Новое значение в TOS пишется
  одновременно через mux.
- При `dpop`: `TOS ← NOS`, `NOS ← SRAM[--DSP]`.

### Размещение данных (ответы на «контрольные» вопросы из задания)

- **Числовой литерал → непосредственная адресация?** Да, через инструкцию `LIT`,
  значение лежит в следующем слове памяти инструкций. Адресации к data-секции не нужно.
- **Литерал → в статическую память?** Только строки (`." ..."`, `S" ..."`). Числа
  никогда — они всегда `LIT`.
- **Раскладка литералов в статической памяти?** Подряд в порядке появления в исходнике.
  Каждый символ — одно слово (`cstr`), терминатор — нулевое слово.
- **Литерал, требующий нескольких слов?** Только строки (N+1 слов). Между ними
  паддинг не нужен, слова и так выровнены.
- **Переменная → регистр?** Никогда явно. Стековая машина не имеет программно-адресуемых
  регистров. Промежуточные значения вычислений живут на data stack.
- **Переменная → статическая память?** Всегда, для `variable name`. Имя при компиляции
  превращается в push адреса этой ячейки. Чтение — `x @`, запись — `value x !`.
- **Процедуры:** компилируются в код-секцию, последняя инструкция — `RET`. Адрес первой
  инструкции = xt этого слова.
- **Прерывания:** в варианте `stream` отсутствуют. Ввод/вывод синхронный, через MMIO.

## Система команд (ISA)

### Особенности процессора

- Стековая машина: операции работают с вершинами data stack (`TOS`, `NOS`).
- Все инструкции — **одно машинное слово 32 бита**. Это сделано RISC-стиле: фиксированная
  длина, простое декодирование. Исключение — `LIT`, которая занимает **2 слова**:
  опкод + 32-битное значение.
- Поток управления:
    - инкремент `PC` после каждой инструкции;
    - условный (`JZ`) и безусловный (`JMP`) переходы; вызовы (`CALL` / `RET`); косвенный
      вызов (`EXECUTE`).
- Память — однопортовая (как требует вариант). Доступ — через **кеш**, на промах +9 тактов.
- I/O — memory-mapped, регион MMIO uncached.

### Набор инструкций

| Mnem | Opcode | Стек до → после | Описание | Тактов* |
|---|---|---|---|---|
| `NOP` | 0x00 | — | пусто | 1 |
| `HALT` | 0x01 | — | остановка модели | 1 |
| **Стек данных** ||||
| `LIT` | 0x10 | → n | push константы (значение в след. слове) | 4 |
| `DUP` | 0x11 | a → a a | дубль вершины | 2 |
| `DROP` | 0x12 | a → | снять | 1 |
| `SWAP` | 0x13 | a b → b a | обмен двух вершин | 2 |
| `OVER` | 0x14 | a b → a b a | копия второй сверху | 3 |
| **АЛУ** ||||
| `ADD` `SUB` `MUL` `DIV` `MOD` | 0x20–0x24 | a b → r | бинарные | 2 |
| `AND` `OR` `INVERT` | 0x27, 0x28, 0x29 | — | побитовые | 2 |
| `EQ` `LT` `GT` | 0x25, 0x26, 0x2A | a b → flag | сравнения (-1 если true, 0 иначе) | 2 |
| **Память (через кеш)** ||||
| `LOAD` (`@`) | 0x30 | addr → val | чтение из памяти | 3+M |
| `STORE` (`!`) | 0x31 | val addr → | запись в память | 3+M |
| **Поток управления** ||||
| `JMP a` | 0x40 | — | безусловный переход на адрес `a` | 1 |
| `JZ a` | 0x41 | flag → | переход если pop == 0 | 2 |
| `CALL a` | 0x42 | — | push PC+1 на RS; PC ← a | 2 |
| `RET` | 0x43 | — | PC ← pop с RS | 2 |
| `EXECUTE` | 0x44 | xt → | call по адресу с вершины DS | 2 |
| **Стек возвратов** ||||
| `>R` | 0x50 | a → ⟨RS: a⟩ | переслать с DS на RS | 2 |
| `R>` | 0x51 | ⟨RS: a⟩ → a | переслать с RS на DS | 2 |
| `R@` | 0x52 | → a | копия вершины RS на DS | 2 |

\* Без учёта промахов кеша и FETCH-а инструкции (он тоже идёт через кеш). M = +9 тактов
на miss, +9 ещё на write-back грязной строки при вытеснении.

**Классификация:** stack ISA, RISC-стиль (фиксированная длина), Neumann, microcoded
ControlUnit.

### Кодирование инструкций (binary)

Бинарное представление — настоящие бинарные файлы (требование варианта); транслятор
параллельно пишет отладочный текстовый дамп.

```text
┌─────────┬─────────────────────────────────────────────────────────────┐
│ 31...24 │ 23                                                        0 │
├─────────┼─────────────────────────────────────────────────────────────┤
│  opcode │                  operand (24 бита)                          │
└─────────┴─────────────────────────────────────────────────────────────┘
```

- `opcode` (8 бит) → 256 возможных опкодов (используем ~30).
- `operand` (24 бита) → адрес перехода для `JMP`/`JZ`/`CALL` (хватает на адресное
  пространство до 16 МБ слов, что покрывает реальные нужды).
- Для инструкций без операнда поле игнорируется (заполняется нулями).
- **Спецслучай `LIT`:** опкод-слово + следующее слово, содержащее 32-битное значение.
  При FETCH процессор увидит `LIT` и в микропрограмме `LIT` сам прочитает ещё одно
  слово и инкрементирует `PC` дважды.

**Пример дампа** (формат как в задании: `<address> - <HEXCODE> - <mnemonic>`):

```text
0  - 40000005 - JMP 5
1  - 10000000 - LIT
2  - 0000002A - <value 42>
3  - 11000000 - DUP
4  - 43000000 - RET
5  - 42000001 - CALL 1
6  - 01000000 - HALT
```

Кроме `.bin` (сырой бинарь) транслятор пишет `.bin.hex` (этот дамп). Это сильно
помогает при отладке и попадает в golden тесты.

## Транслятор

CLI: `translator.py <source.fth> <target.bin>`

Этапы (функция `translate`):

1. **Tokenize.** Разбить исходник на токены: слова, разделённые whitespace; обработать
   строковые литералы (`." ..."`, `S" ..."`), комментарии (`( ... )`, `\ ...`).
2. **Compile (single-pass).** Идём по токенам, для каждого:
    - число → emit `LIT` + value слово;
    - встроенное слово → emit его опкод;
    - имя пользовательского слова → emit `CALL <addr_of_word>`;
    - `: name` → начало определения, запоминаем `name → current_addr` в словаре;
    - `;` → emit `RET`, закрываем определение;
    - `if` → emit `JZ ?`, push текущий адрес на compile-time stack для backpatch;
    - `else` → emit `JMP ?`, patch предыдущий `JZ` на адрес-после-jmp, push новый;
    - `then` → patch последний адрес из стека на текущий;
    - `begin` → push current_addr;
    - `until` → emit `JZ <addr_from_stack>`;
    - `variable x` → выделить слово в data-секции, в словарь `x → push <addr>`;
    - `constant N x` → в словарь `x → LIT N`;
    - `' word` → emit `LIT <addr_of_word>`;
    - `." text"` → разместить строку в data-секции (cstr), emit `LIT <addr> CALL print_cstr`.
3. **Layout.** Собрать финальный образ: сначала код (с прыжком на main в начале), затем
   data-секция.
4. **Emit.** Записать `.bin` (сырые байты) и `.bin.hex` (текстовый дамп).

Backpatching реализуется так же, как в референсе с `[`/`]`: храним placeholder в массиве
инструкций и заменяем его, когда узнаём целевой адрес.

## Модель процессора

CLI: `machine.py <code.bin> <input.txt>`

Логически состоит из четырёх частей:

```text
+--------------------+    signals    +-----------+
|    ControlUnit     |==============>|  DataPath |
|    (microcoded)    |<==(flags)=====|           |
+--------------------+               +-----------+
       |                                  |
       | mem req                          | mem req
       v                                  v
+--------------------+               +-----------+
|       Cache        |==============>| MainMemory|
| (direct-mapped,    |               |  + MMIO   |
|  write-back)       |               | (uncached)|
+--------------------+               +-----------+
```

Все четыре — отдельные классы в Python; каждый имеет узкое API и тестируется
изолированно. Cache и MMIO живут «под» MainMemory, ControlUnit ходит в память
только через Cache.

### DataPath

```text
                    LatchPc/JmpMux                   ┌─────────────────┐
                          │                          │   MainMemory    │
                          v                          │     +  MMIO     │
                     ┌─────────┐    addr            │                 │
                     │   PC    │─────────┐          └───────▲─────────┘
                     └─────────┘         │                  │
                          ▲              v                  │
                          │       ┌────────────┐            │
                          │       │   Cache    │◀──── addr,op,data
                          │       │ (write-back│
                          │       │  direct-   │            │ done
                          │       │   mapped)  │────────────┘ value
                          │       └─────┬──────┘
                          │             │ value
                          │             v
                     ┌─────────┐  ┌─────────┐  ┌──────────┐
                     │   IR    │  │   MAR   │  │   MDR    │
                     └────┬────┘  └─────────┘  └────┬─────┘
                          │ opcode/operand              │
                          v                              v
                     ┌──────────┐               ┌────────────────┐
                     │  Decode  │               │  TOS  │  NOS   │◀── DStack SRAM (256)
                     │ opc→mPC  │               └───┬───┴───┬────┘
                     └────┬─────┘                   │       │
                          │                         v       v
                          v                       ┌───────────┐
                     ┌──────────┐                 │    ALU    │──► Z, N
                     │   mPC    │                 └─────┬─────┘
                     └────┬─────┘                       │
                          │ addr                        v
                          v                       ┌──────────┐
                     ┌──────────┐    signals      │ Mux в TOS│
                     │ mProgram │────────────────▶│ Mux в NOS│
                     │   ROM    │                 │ Mux в PC │
                     └──────────┘                 │ ...      │
                                                  └──────────┘
                                                       │
                                                       v
                                                  ┌──────────┐
                                                  │   TORS   │◀── RStack SRAM (256)
                                                  └──────────┘
```

Реализован в классе `DataPath`. Его методы соответствуют сигналам, каждый исполняется
за один такт. Корректность последовательности сигналов — ответственность `ControlUnit`.

**Регистры:** `PC`, `IR`, `TOS`, `NOS`, `DSP`, `TORS`, `RSP`, `MAR`, `MDR`. Плюс
флаги `Z`, `N`, обновляемые ALU.

**Сигналы DataPath:**

- `latch_pc(sel)` — обновить PC. `sel`: `Next` (+1), `Jmp` (из IR.operand), `Jz`
  (Jmp если zero, иначе Next), `FromTos` (для EXECUTE).
- `latch_ir()` — защёлкнуть MDR в IR.
- `latch_mar(sel)` — обновить MAR. `sel`: `FromPc`, `FromTos`.
- `latch_mdr_from_mem()` — защёлкнуть результат памяти в MDR.
- `latch_mdr_from_nos()` — защёлкнуть NOS в MDR (для STORE).
- `mem_request(op)` — отправить запрос в кеш. `op`: `Read`, `Write`.
- `latch_tos(sel)` — `FromAlu`, `FromMdr`, `FromNos`, `FromIrOperand`, `FromMem` (для MMIO).
- `latch_nos(sel)` — `FromTos`, `FromDsSram`.
- `dstack_push()` — `SRAM[DSP++] ← NOS`.
- `dstack_pop()` — `NOS ← SRAM[--DSP]` (одновременно с `latch_nos`).
- `latch_tors(sel)` — `FromTos`, `FromPc` (для CALL), `FromRsSram`.
- `rstack_push()` — `RSRAM[RSP++] ← TORS`.
- `rstack_pop()` — `TORS ← RSRAM[--RSP]`.
- `alu_op(op)` — установить операцию ALU; результат идёт в `latch_tos(FromAlu)`.

**Флаги:**

- `Z` — `TOS == 0` (обновляется при `alu_op`).
- `N` — `TOS < 0`.

### Кеш

Отдельный модуль между ControlUnit и MainMemory.

**Параметры:**

- direct-mapped;
- **4 строки × 4 слова** (16 слов всего);
- write-back + write-allocate;
- hit = 1 такт, miss = +9 тактов, dirty eviction = ещё +9 тактов.

**Разбиение адреса (32 бита):**

```text
┌──────────────────────┬───────┬──────┐
│         tag          │ index │offset│
│       28 бит         │ 2 бит │ 2 бит│
└──────────────────────┴───────┴──────┘
```

**Поведение:**

- `request(addr, op, data=None)` — запустить операцию (read/write).
- На каждом такте — `tick()`. Внутри: если в процессе miss-обработки — крутит счётчик
  ожидания; когда отработал — выставляет `done=True`.
- `is_done()` / `response()` — проверка готовности и получение результата (для read).
- **MMIO bypass:** если адрес `>= 0xFFFFFFF0` — запрос не идёт через таблицу строк,
  а сразу проксируется в MMIO-контроллер, готовность за 1 такт.

**Логирование:** каждый запрос пишет в журнал строку вида:

```text
TICK: 42 cache READ  addr=0x00000100 → HIT  line=0
TICK: 51 cache READ  addr=0x00000200 → MISS line=0 evict=clean (stall 9)
TICK: 70 cache WRITE addr=0x00000300 → MISS line=0 evict=DIRTY wb=0x00000100 (stall 18)
```

### Микрокодированный ControlUnit

Архитектурно (схема в стиле OCaml-референса, см. `microcoded.ml`):

```text
                            +------------------(+1)-------+
                            |                             |
                            |    latch_pc                 |
                            |                  |          |
                            |   +-----+        v          |
                            +-->|     |     +---------+   |    +---------+
                                | MUX |---->|   PC    |---+--->| Memory  |
                            +-->|     |     +---------+        | (Cache) │
                            |   +-----+                        +---------+
                            |      ^                                |
                            |      | sel_pc                         | current
                            |      |                                | instruction
                            +---------------(select-operand)--------+
                                   |                                |
                                   |                       (opcode) v
                                +-----+                    +---------------+
                                | MUX |<-- jmp_sel         | opcode -> mPC |
                                +-----+                    +---------------+
                                 ^   ^                            |
                                 |   |                            | mPC address
                                 |   |                            |
                                 |   |                       0--+ | +-----+
                                 |   |                          | | |     |
                                 |   |                          v v v     |
                                 |   |                         +-----+    |
                                 |   |          sel_mPC ───────►│ MUX │  (+1)
                                 |   |                         +-----+    |
                                 |   |                            |       |
                                 |   |                            v       |
                                 |   |        +----------+    +-----+     |
                                 |   +--------| mProgram |◀───|  mPC│─────+
                                 |            +----------+    +-----+
                                 |                 |
                                 |                 | signals (list)
                                 |                 v
                                 |  flags    +----------+
                                 +───────────|          |
                                             | DataPath |─── mem req ──► Cache
                                             |          |
                                             +----------+
```

Реализован в классе `ControlUnit`.

**Регистры CU:** `mPC` (адрес микроинструкции).

**Принцип работы (один такт):**

1. Взять `mProgram[mPC]` — это **список сигналов** (одна микроинструкция).
2. Применить все сигналы из списка к DataPath / себе (в порядке записи).
3. `tick()` — увеличить модельное время на 1.
4. Если на этом такте кеш «занят» (запрос ещё не отработан) — **сигналы, требующие
   `mem_done`, блокируют переход mPC**: значит этот же набор сигналов исполнится и
   на следующем такте. Технически это спин-микроинструкция (см. ниже).

**Сигналы ControlUnit** (в отличие от сигналов DataPath, эти меняют состояние CU,
а не данных):

- `LatchMpc(sel)` — `Zero` (вернуться в FETCH), `Opcode` (декод по IR.opcode),
  `Next` (mPC+1).
- `Halt` — поднять флаг остановки.

**Декодер opcode → mPC:** обычный `dict[Opcode, int]`, заполняемый при компиляции
микропрограммы. Прямая аналогия с `mpc_of_opcode` из OCaml-референса.

### Микропрограмма

Микропрограмма — **массив списков сигналов**, как в OCaml-референсе. Каждый элемент —
одна микроинструкция. Сигнал — экземпляр enum-подобного типа с возможным аргументом
(`LatchPc(Jmp)`, `LatchDataAddr(Left)` и т.д.).

```python
mprogram: list[list[Signal]] = [
    # 0: FETCH
    [LatchMar(FromPc), MemRequest(Read), LatchMpc(Next)],
    # 1: FETCH (wait + capture IR + decode)
    [LatchIr(), LatchPc(Next), LatchMpc(Opcode)],

    # ─── ADD (opcode 0x20) ───
    # 2: вычислить TOS+NOS, записать в TOS, снять NOS
    [AluOp(Add), LatchTos(FromAlu), DStackPop(), LatchMpc(Zero)],

    # ─── DUP (opcode 0x11) ───
    # 3: push текущий TOS, NOS становится копией TOS
    [DStackPush(), LatchNos(FromTos), LatchMpc(Zero)],

    # ─── LIT (opcode 0x10) ───
    # 4: прочитать следующее слово как значение
    [LatchMar(FromPc), MemRequest(Read), LatchMpc(Next)],
    # 5: записать значение в TOS, NOS получает старый TOS
    [DStackPush(), LatchNos(FromTos), LatchTos(FromMdr), LatchPc(Next), LatchMpc(Zero)],

    # ─── LOAD (opcode 0x30) ───
    # 6: MAR ← TOS, запрос
    [LatchMar(FromTos), MemRequest(Read), LatchMpc(Next)],
    # 7: TOS ← MDR (если кеш не готов на этом такте — мы здесь и зависнем)
    [LatchTos(FromMdr), LatchMpc(Zero)],

    # ─── JZ addr (opcode 0x41) ───
    # 8: проверить TOS, выбрать Jz, снять TOS
    [LatchPc(Jz), DStackPop(), LatchMpc(Zero)],

    # ─── CALL addr (opcode 0x42) ───
    # 9: TORS ← PC+1 (= уже инкрементированный PC), push на RS
    [LatchTors(FromPc), RStackPush(), LatchPc(Jmp), LatchMpc(Zero)],

    # ─── RET (opcode 0x43) ───
    # 10: PC ← TORS, pop с RS
    [LatchPc(FromTors), RStackPop(), LatchMpc(Zero)],

    # ... и так далее на каждый opcode
]
```

**Карта `opcode → начало микропрограммы`** живёт в коде как функция/словарь:

```python
mpc_of_opcode = {
    Opcode.HALT:   ...,
    Opcode.ADD:    2,
    Opcode.DUP:    3,
    Opcode.LIT:    4,
    Opcode.LOAD:   6,
    Opcode.JZ:     8,
    Opcode.CALL:   9,
    Opcode.RET:   10,
    # ...
}
```

**Ожидание готовности кеша.** Если микроинструкция содержит сигнал, требующий
`MemRequest`, и кеш ещё не готов с предыдущего запроса — `ControlUnit` **не
продвигает mPC** на следующем такте, оставаясь на текущей микроинструкции. С точки
зрения железа это эквивалентно тому, что регистры просто не защёлкиваются (clock-gating
ввиду `stall`-сигнала от кеша). В журнале это видно как несколько подряд тиков с
одним и тем же `mPC` и пометкой `cache busy`.

## Расширяемость: как добавить новую инструкцию

Архитектура спроектирована так, чтобы добавление инструкции через **уже существующие
сигналы** не требовало правок в DataPath, Cache, ControlUnit или формате микроинструкции.

### Пример: добавляем `BGE` (Bigger-or-Equal: `a b → -1 if a≥b else 0`)

**Шаг 1.** В `isa.py` — добавить опкод и мнемонику:
```python
class Opcode(IntEnum):
    ...
    BGE = 0x2B
```

**Шаг 2.** В `datapath.py` (или `alu.py`) — добавить случай в ALU:
```python
class AluOp(Enum):
    ...
    Ge = "ge"

# и обработка:
case AluOp.Ge: result = -1 if a >= b else 0
```

**Шаг 3.** В `microcode.py` — добавить микропрограмму и привязку:
```python
# в mprogram, по новому индексу N:
[AluOp(Ge), LatchTos(FromAlu), DStackPop(), LatchMpc(Zero)],

# в mpc_of_opcode:
Opcode.BGE: N,
```

**Шаг 4.** В трансляторе — слово Forth → опкод:
```python
BUILTINS = {
    ...
    ">=": Opcode.BGE,
}
```

**Шаг 5.** Golden тест на `>=` (1 файл YAML).

**Что НЕ меняется:** DataPath (новых сигналов не появилось — используется существующий
`AluOp` с новым значением, существующие `LatchTos(FromAlu)`, `DStackPop()`, `LatchMpc(Zero)`),
ControlUnit, Cache, формат микроинструкции, бинарный формат инструкции, CLI.

### Сценарий: инструкция-композиция

Например, `2DUP` (`a b → a b a b`) — не делается одной микроинструкцией (за такт можно
писать в один регистр). Решается **последовательностью существующих сигналов** в
несколько микроинструкций. Никаких новых сигналов не появляется — это и есть смысл
микрокода.

### Сценарий: нужен новый сигнал / новый источник для mux

Например, инструкция `DEPTH` (положить размер data stack на стек) требует `DSP` в качестве
источника для TOS. Это — **изменение DataPath**: добавить новый вариант `LatchTos(FromDsp)`,
добавить ветку в реализации mux. Это **архитектурное** изменение, но цена низкая:
один новый вариант enum + один новый case в `latch_tos`.

### Что специально сделано ради расширяемости

- Сигналы — **именованные перечисления**, легко добавлять варианты.
- Микропрограмма — **данные**, не код: добавление инструкции = добавление строк в массив
  и записи в `mpc_of_opcode`.
- Декодер opcode → mPC — отдельный dict, новые опкоды цепляются без правки логики CU.
- Все mux'ы централизованы в DataPath, добавить новый вход — одно место.

## Структура исходного кода

В каноне референса (минимум файлов, всё плоское):

```text
.
├── README.md                  # отчёт по варианту (заполняется по этому документу)
├── ARCHITECTURE.md            # этот файл
├── pyproject.toml             # poetry + ruff + mypy + pytest + pytest-golden
├── Makefile                   # format / lint / test / test-update-golden
├── .github/workflows/ci.yml   # CI
│
├── isa.py                     # Opcode, кодирование/декодирование, формат бинаря
├── microcode.py               # сигналы, mprogram, mpc_of_opcode
├── translator.py              # Forth → bin (+ hex дамп)
├── cache.py                   # Cache (отдельно ради тестируемости)
├── machine.py                 # DataPath + ControlUnit + simulation + main
├── golden_test.py             # один тестовый файл, читает golden/*.yml
│
├── examples/                  # примеры программ
│   ├── hello.fth
│   ├── cat.fth
│   ├── hello_user_name.fth
│   ├── sort.fth
│   ├── double_precision.fth
│   ├── prob1.fth              # Euler #4
│   ├── execute_demo.fth       # демо xt / execute (особенность Forth)
│   └── cache_demo.fth         # демо эффекта кеша
│
└── golden/                    # YAML-конфиги golden-тестов
    ├── hello.yml
    ├── cat.yml
    ├── hello_user_name.yml
    ├── sort.yml
    ├── double_precision.yml
    ├── prob1.yml
    ├── execute_demo.yml
    └── cache_demo.yml
```

### Почему такая «плоская» структура

Это **сознательное решение, повторяющее канон**:

- Референс [ryukzak/brainfuck](https://github.com/ryukzak/brainfuck) держит весь код
  в `isa.py`, `translator.py`, `machine.py`. Никаких пакетов, никаких `__init__.py`.
- Задание прямо предупреждает: «не стоит заниматься overengineering-ом… десяток файлов
  раскиданным по 5 папкам».
- Тестируемость отдельных модулей достигается **через doctest и через прямой импорт**
  (см. `>>> dp = DataPath(...)` в docstring референса), а не через дробление
  кодовой базы.

Если по ходу реализации какой-то файл вырастает выше ~500 строк — выделяется
подмодуль (например, `lexer.py` отдельно от `translator.py`). До этого — плоско.

### Тестируемость каждого модуля

- **`isa.py`** — doctest на `to_bytes`/`from_bytes`/`to_hex` (round-trip).
- **`cache.py`** — отдельный pytest-файл, серия запросов с проверкой счётчиков hits/misses
  и eviction'ов. Mock'аем `MainMemory` как `list[int]`.
- **`machine.DataPath`** — doctest'ы внутри методов-сигналов: завернул в `DataPath`,
  дёрнул сигналы, проверил поля. Так делает референс (см. doctest у `signal_wr`).
- **`machine.ControlUnit`** — doctest на короткой программе: «загрузили `[LIT, 5, HALT]`,
  прогнали — TOS=5».
- **`microcode.py`** — отдельный тест: для каждого опкода стартуем mPC на его адрес,
  прогоняем сигналы на голом DataPath, проверяем итоговые регистры. Это **самое
  важное** для уверенности в реализации варианта `mc`.
- **`translator.py`** — unit-тесты на коротких snippet'ах + всё остальное через golden.

## Тестирование

Подход — golden tests + doctest, как в референсе.

### Формат golden-теста (один YAML на алгоритм)

```yaml
in_source: |-
  : square dup * ;
  5 square emit
in_stdin: ""
out_code: !!binary |
  <base64 бинаря>
out_code_hex: |-
  0  - 40000005 - JMP 5
  1  - 11000000 - DUP
  2  - 22000000 - MUL
  3  - 43000000 - RET
  4  - 00000000 - NOP
  5  - 10000000 - LIT
  6  - 00000005 - <value 5>
  7  - 42000001 - CALL 1
  8  - ...
out_stdout: |
  source LoC: 2 code instr: 9
  ============================================================
  
  ticks: 47
  cache: hits=12 misses=3 writebacks=0
out_log: |-
  DEBUG machine:simulation  TICK:   0 PC: 0 mPC: 0 TOS: 0 NOS: 0 ...  FETCH
  DEBUG cache:request       READ addr=0x00000000 → MISS line=0 (stall 9)
  ...
```

Поля повторяют формат референса (`in_source`, `in_stdin`, `out_code`, `out_code_hex`,
`out_stdout`, `out_log`), плюс мы можем добавлять `cache: ...` для демонстрационных тестов.

### Обязательные алгоритмы (по заданию)

1. `hello` — Hello, World!
2. `cat` — эхо ввода в вывод (потенциально бесконечный ввод).
3. `hello_user_name` — спросить имя, поздороваться.
4. `sort` — отсортировать числа из ввода.
5. `double_precision` — арифметика 64 бит на 32-битных словах (пары слов hi/lo).
6. `prob1` — Euler problem 4 (largest palindrome product).
7. **+ демонстрационные тесты** варианта:
    - `execute_demo` — особенность Forth (xt + EXECUTE).
    - `cache_demo` — два прохода по массиву, видны hits/misses/writebacks.
    - `microcode_inspect` — короткая программа с журналом до уровня микроинструкций.

### Golden-инфраструктура

Один файл `golden_test.py`, копия паттерна из референса:

```python
@pytest.mark.golden_test("golden/*.yml")
def test_translator_and_machine(golden, caplog):
    # запуск translator.main → machine.main во временной папке
    # сравнение out_code / out_code_hex / out_stdout / out_log с golden.out
```

Обновление эталонов: `make test-update-golden`.

### CI

`.github/workflows/ci.yml`:

- `poetry run ruff format --check .`
- `poetry run ruff check .`
- `poetry run mypy --strict isa.py microcode.py translator.py cache.py machine.py`
- `poetry run pytest -v`

Конфигурация ruff копируется из референса (`pyproject.toml`), включая все «суровые»
правила. Любое отключение правила — с комментарием.

## Резюме

Эта архитектура:

- **Повторяет структуру референса**, на который опирается задание. Преподавателю
  привычно проверять, и любой пункт оценки (golden tests / CI / lint / читаемость
  схем / реалистичность микрокода / реализация варианта) закрыт по шаблону, который
  уже принят как «правильный».
- **Адаптирована под наш вариант** (`forth | stack | neum | mc | tick | binary |
  stream | mem | cstr | prob1 | cache`): добавлены `cache.py` и `microcode.py`,
  карта памяти расширена под MMIO в стиле `mem`, ISA выстроена вокруг стека и cstr.
- **Тестируема по модулям**: каждый из четырёх рабочих компонентов (DataPath, Cache,
  ControlUnit, mProgram) имеет узкое API и тестируется в изоляции — через doctest
  и через отдельные pytest-файлы — до того, как собирается полная цепочка.
- **Расширяема**: добавление новой ALU-инструкции — три декларативных правки;
  добавление инструкции-композиции — несколько строк в микропрограмме без новых
  сигналов; всё это без правок в сердце симулятора.
